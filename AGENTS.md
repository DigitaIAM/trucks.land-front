# trucks.land-front — agent guide

## Quick start

```sh
npm install
npm run dev          # Vite dev server on :5173
npm run check-build  # type-check + build
npm test             # vitest run
npm run lint         # eslint . --fix
npm run format       # prettier --write src/
```

## Non-obvious scripts

| Command               | What it does                        | Why it matters                                                                                                                  |
|-----------------------|-------------------------------------|---------------------------------------------------------------------------------------------------------------------------------|
| `npm run check-build` | `vue-tsc --build` then `vite build` | `npm run build` skips type-checking (runs `build-only` in parallel via `run-p`). Use `check-build` before CI-like verification. |
| `npm run type-check`  | `vue-tsc --build`                   | Runs across project references (`tsconfig.app.json` + `tsconfig.node.json`), which is stricter than a single `tsc` pass.        |
| `npm run deploy`      | `gh-pages -d build`                 | **Stale.** CI uses `./dist`; this script outputs to `dist/` but deploys from `build/`. Don't use it — rely on CI.               |

## Dev server quirks

- CORS origin whitelist: `http://localhost:5173`, `https://api.zeptomail.com`
- Proxy at `/zeptomail` → `https://api.zeptomail.com` (rewrites path prefix)
- `.env` contains Supabase credentials (`VITE_SUPABASE_URL`, `VITE_SUPABASE_KEY`); these are secrets in CI

## Auto-imports (no explicit imports needed)

- **Vue APIs** — `ref`, `computed`, `watch`, `reactive`, `defineComponent`, etc.
- **Vue Router** — `useRouter`, `useRoute`, `RouterLink`, `RouterView`
- **VueUse** — `useColorMode`, `computedAsync`, `debouncedWatch`, `useLocalStorage`, etc.
- **Stores** — `useXxxStore()` from `src/stores/` (e.g., `useAuthStore()`, `useOrdersStore()`)
- **Composables** — any export from `src/composables/`
- **Models** — any export from `src/models/`
- **Vue macros** — `$ref`, `$computed`, `$shallowRef` etc.
- All **components** under `src/components/` are globally registered (PascalCase, e.g. `<Button>`, `<Card>`, `<Modal>`)

## Architecture

- **Vue 3 SPA** with file-based routing (`src/pages/` structure maps to URLs) and layouts (`src/layouts/`, applied via
  `<route lang="yaml">` blocks in page files)
- **Pinia** stores in `src/stores/` — auto-imported, naming: `useXxxStore`
- **Supabase** for auth and real-time DB (`src/composables/use-supabase.ts` provides `supabase` client + channel
  subscriptions for `orders`, `order_stages`, `order_events`)
- **DaisyUI** (Tailwind CSS v4) — widgets in `src/components/widgets/` mirror DaisyUI classes (e.g. `<Button primary>`,
  `<Card>`, `<Modal>`)
- **VueDatePicker** from `@vuepic/vue-datepicker` — registered globally as `<VueDatePicker>` in `src/main.ts`
- **SVGs** in `<template>` importable as Vue components via `vite-svg-loader`
- **Icons** via `unplugin-icons` (Iconify) — `<Icon name="mdi:home" />`
- `supabase` client injected as `app.provide('supabase', supabase)` in `main.ts`; also importable from
  `@/composables/use-supabase`

## Lint / format

**Order: lint → format** (ESLint with `@vue/eslint-config-prettier/skip-formatting` means ESLint does not format —
Prettier handles that separately).

VSCode does this automatically on save (`editor.codeActionsOnSave.source.fixAll` + `editor.formatOnSave` with Prettier
as default formatter).

- `npm run lint` — ESLint flat config (`eslint.config.ts`), targets `**/*.{ts,mts,tsx,vue}`, ignores `dist/`,
  `dist-ssr/`, `coverage/`
- `npm run format` — Prettier with no semicolons, single quotes, 100 print width, only `src/`

## Type-check

`npm run type-check` runs `vue-tsc --build` using project references (`tsconfig.json` → `tsconfig.app.json` +
`tsconfig.node.json`). This is the single truth for type errors — do not rely on `tsc` directly.

## Tests

Vitest (v4) — run with `npm test` (or `npm run test:watch`). Tests live in `src/composables/__tests__/` and use global
mocks from `vitest.setup.ts`.

| Test file                                | Tests | What it covers                                                                                                                                                                     |
|------------------------------------------|-------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `use-employee-report-calculator.test.ts` | 22    | `getWorkingDaysInRange` (pure, 7 cases), `loadOrdersInProgress`, `loadUnpaidOrders`, `loadUnpaidSettlements`, `calculateEmployeeReport` (integration with supabase mocks, 5 cases) |
| `use-owner-report-calculator.test.ts`    | 12    | `loadOwnerPayments`, `loadOwnerExpenses`, `calculateOwnerReport` (search filtering, payout calc, sorting)                                                                          |

**Mock strategy:** `global.supabase.from` returns a chainable thenable query builder; per-table return data is set via
`mockTableData` object. Pinia stores (`useUsersStore`, `useOwnersStore`) are auto-mocked in `vitest.setup.ts`. Override
per test with `mockImplementationOnce`.

## CI / deploy

- GitHub Actions (`.github/workflows/deploy.yml`): triggered on push to `main`
- `npm ci` → `npm run build` (env vars from secrets) → `cp dist/index.html dist/404.html` → upload to GitHub Pages
- Supabase URL/key: `VITE_SUPABASE_URL`, `VITE_SUPABASE_KEY` as secrets

## Gotchas

- `src/views/` exists but is **empty** — not used; pages live in `src/pages/`
- The README is a generic Vite starter template — ignore it for project-specific guidance
- All stores, composables, and models are auto-imported; do not manually import them unless you need a type only
- `src/utils/` is **not** auto-imported — import from it explicitly
- Pinia stores use `acceptHMRUpdate` for hot reload support
- `typed-router.d.ts` should be included in tsconfig for typed route names
